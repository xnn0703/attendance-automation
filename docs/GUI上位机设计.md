# 南京六部考勤 GUI 上位机 — 设计文档（v0.1）

> **状态（2026-06-03）：Phase 1 MVP + Phase 2（彩色预览 / 双击改判即时重算 / 决策存取 / 输入校验）+ Phase 3
> （PyInstaller 打包 `app/dist/mingxu.exe`）均已实现（方案 A，对 2026-05 逐格零回归）。**
> 下文为设计依据；已实现见 `app/engine.py`、`app/gui.py`、`app/build_exe.bat`。

> 目的：把已固化、已验证的考勤工作流（skill `nanjing-kaoqin`）做成一个**桌面上位机**：可视化驱动，
> 自动完成能自动的部分，对两个真正需要人工判断的点弹交互，协作生成《初表》与《考勤》。
> 本文档供评审；评审通过后再编码。

---

## 1. 目标与范围

- **做什么**：本地单机桌面 app。选一个含三张输入表的目录 → 一步步跑完工作流 → 出《南京六部（初表）》与《南京六部（考勤）》。
- **半自动**：能自动的全自动（解析、日历、对齐、加班、迟到早退、缺卡判定的"建议值"、外勤、全勤、着色、写表）；
  仅在两个点弹交互——①离职且打卡<7天保留谁；②工作日打卡<2次逐条归类（缺卡/公出/未出勤）。
- **MVP 不做**：联网、账号权限、多部门、历史数据库。先把"换月一键半自动出表"跑顺。

## 2. 技术栈与前提（关键）

- **选定**：Python + Qt（具体用 **PySide6**——Qt 官方绑定，LGPL 许可更省心；不建议 PyQt 的 GPL）。
- ⚠️ **前提（必须先解决）**：本机当前 `python` 是 Microsoft Store 占位版，**无 `pip`**，装不了任何包。
  走 Python 路线需先：
  1. 安装真 Python 3.11+（python.org；勾 *Add to PATH*；并在"管理应用执行别名"里关掉 python 的 Store 别名）；
  2. `pip install PySide6 openpyxl`。
- **本机已确认可用（零安装）**：.NET Framework 的 **WPF / WinForms**（即附录 B 的备选不需要任何安装）。

## 3. 架构（两种，评审二选一）

引擎（规则+读写）与界面（交互+可视化）分层。GUI 只调引擎，不重复规则逻辑。

### 方案 A（推荐）：全 Python —— Qt 界面 + openpyxl 引擎
- 装好真 Python 后 `openpyxl` 可用 → **用 Python 重写考勤引擎**（解析/日历/核算/着色/写 xlsx），与 Qt 同语言。
- 引擎写成**纯函数模块**（不依赖 GUI），便于单测。
- **回归兜底**：以现有 PowerShell 引擎产出的 2026-05《考勤》为**金标准**，Python 引擎跑同月数据后**断言逐格一致**
  （加班数值 / 字体色 / 底色 / 全勤），达成零回归再往下做。规则已在 `references/rules.md` 固化，照搬即可。
- 优点：单语言、`openpyxl` 处理样式比手改 XML 稳健、最易维护与产品化。
- 代价：重写引擎（但有金标准 + 现成规则，风险可控）。

### 方案 B（低风险过渡）：Qt 界面 + 复用现有 PowerShell 引擎
- Qt 用 `subprocess` 调 `kq_run.ps1`；为便于程序消费，给脚本加一个 **JSON 输出模式**（而非给人看的文本）。GUI 写配置 txt。
- 优点：引擎零改动、零回归、最快见界面。
- 缺点：双语言、PS↔Python 子进程 + JSON 桥接，长期不如 A 干净。

```
方案A:  [Qt 界面] ──直接调用──> [Python 引擎(openpyxl)] ──> xlsx
方案B:  [Qt 界面] ──subprocess+JSON──> [kq_run.ps1 引擎] ──> xlsx
```

> 建议：若目标是"长期产品化" → **A**；若只想尽快看到能点的界面 → **B**（之后可平滑迁到 A）。

## 4. 界面设计（5 步向导）

主窗口：左侧步骤导航（①…⑤）＋右侧内容区＋底部**日志/状态栏**＋顶部当前**数据目录**。长任务放后台线程（`QThread`），界面不卡，配进度条。

### ① 选择数据目录
- `QFileDialog` 选文件夹；自动按文件名识别三张表（原表/花名册/调班表），逐个显示 ✓/✗；
- 解析并显示：年月、人数、调班表是否含网格。缺表或命名不符给红字提示。`[下一步]`。

### ② 整理 → 《初表》（对应 prep）
- `[运行整理]` → 展示三块：**纳入名单**（工号/姓名/职位，表格）、**删除·非本部门**（列表）、
  **离职且打卡<7天候选**（带勾选框，默认不勾＝删除）。
- 用户勾选要保留者 → `[应用并生成初表]`（保留者只算到离职日）。✓ 后可进下一步。

### ③ 归类 → 人工定性（对应 worklist，**核心交互**）
- `[列出待定项]` → 一张可编辑表 `QTableView`：每个"工作日打卡<2次"案例一行
  （姓名 / 日期 / 星期 / 打卡内容 / 外勤标记 / 打卡次数），**最后一列是下拉**（缺卡/公出/未出勤），**预填建议**
  （0 次→未出勤、1 次→缺卡）。连续空白、外勤单卡等"疑似公出"行高亮提示。
- 顶部可"全部按建议"，再逐行微调。`[保存归类]`。

### ④ 配置（通常沿用默认）
- 表单：不计加班名单 / 严格迟到阈值（人:时刻）/ 只红字迟到名单，预填南京六部当前口径。可加载上月配置。

### ⑤ 生成 → 《考勤》
- `[生成考勤]` → 汇总表（每人加班合计、是否全勤）＋ `[打开考勤表]`。
- （Phase 2）内嵌彩色预览，可单格改判后即时重算。

## 5. 数据流与接口

- 交互产物 = 三个"决策对象"：保留名单、归类映射`{工号|日→类别}`、人员配置。
- 方案 A：决策对象作为参数**直接传给 Python 引擎函数**（内存，无需中间 txt）。
- 方案 B：决策对象**写成 `kq_keep.txt`/`kq_classify.txt`/`kq_config.txt`**，再调 `kq_run.ps1`。
- 引擎对外暴露三个动作：`prep(dir, keep) → 初表/候选`、`worklist(dir) → 待定项列表`、`build(dir, classify, config) → 考勤/汇总`。

## 6. 引擎模块拆分（方案 A）

| 模块 | 职责 |
|------|------|
| `io_xlsx` | 读（inlineStr / 共享串两种）、写（openpyxl 着色/插行） |
| `roster` | 花名册解析、按姓名对齐、重名消歧、在岗区间 |
| `calendar` | 调班表左网格(N班/N休) + 右名单(原/现调日期) → 每人每日 班/休 |
| `punch` | 打卡解析：时间点、外勤、首末次、跨午夜 |
| `rules` | 加班 / 迟到早退 / 缺卡公出未出勤 / 外勤 / 全勤（**纯函数，配置驱动**） |
| `render` | 着色（红字/红绿蓝紫底）、插「加班时长」行、样式表 |
| `pipeline` | 编排 prep / worklist / build |
| `gui` | Qt 层，只调 pipeline，不含规则 |

## 7. 异常与边界

- 三表缺失/命名不符、调班表网格异常、原表样式模板与预期不符（如 fonts 数变化）→ 明确报错而非静默出错。
- 长任务后台线程，界面不冻结。
- 解析/规则异常在日志区给出可读信息（哪个人/哪天/哪条规则）。

## 8. 分期

- **P1 MVP**：5 步向导跑通；方案 A 引擎对 5 月数据零回归（或方案 B 桥接）。
- **P2**：彩色预览、单格改判即时重算、按月配置存取、输入校验。
- **P3**：PyInstaller 打包成 `.exe`（免装 Python 分发）、历史多月、可选「应出勤天数」列、日志导出。

## 9. 待决策（评审拍板）

1. **是否同意安装真 Python**（+ PySide6 + openpyxl）？若不同意 → 走**附录 B**（PowerShell + WPF，零安装）。
2. **引擎方案 A（Python 重写，金标准兜底）还是 B（复用 PS + JSON 桥接）？** —— 推荐 A。
3. UI 是否有品牌/风格要求（标题、配色、图标）？
4. 产出是否需要新增「应出勤天数」等列？
